on writing clean code

published: 2025.06.10 | reading time: ~5 minutes
tags: [programming, best-practices, maintainability]

clean code isn't just about making your program work—it's about making it work in a way that doesn't make future you (or your teammates) want to quit programming altogether.

the golden rule: code is read more than it's written

here's a uncomfortable truth: you'll spend more time reading code than writing it. that function you wrote six months ago? good luck understanding what it does if you named your variables x, temp, and thing.

consider these two approaches: 

// approach 1: cryptic
function calc(x, y, z) {
    return (x * y * z) / 100;
}

// approach 2: clear
function calculateMonthlyInterest(principal, rate, months) {
    return (principal * rate * months) / 100;
}

both functions do the same thing, but only one tells you what it's actually doing without requiring a computer science degree to decipher.

naming things properly

there are only two hard things in computer science: cache invalidation and naming things. here's how to tackle the latter:

"programs must be written for people to read, and only incidentally for machines to execute." — harold abelson

variables should tell a story

  • userAge instead of age
  • isLoggedIn instead of flag
  • maxRetryAttempts instead of limit

functions should be verbs

  • getUserById() instead of user()
  • validateEmail() instead of check()
  • formatCurrency() instead of format()

keep functions small and focused

a function should do one thing and do it well. if your function is longer than your screen, it's probably doing too much. unix philosophy applies here: small, composable pieces that work together. 

// instead of one giant function
function processUserData(userData) {
    // 50 lines of validation
    // 30 lines of transformation  
    // 20 lines of saving
}

// break it down
function validateUserData(userData) { /* ... */ }
function transformUserData(userData) { /* ... */ }
function saveUserData(userData) { /* ... */ }

function processUserData(userData) {
    const validData = validateUserData(userData);
    const transformedData = transformUserData(validData);
    return saveUserData(transformedData);
}

comments should explain why, not what

your code should be self-documenting. comments are for explaining the reasoning behind decisions, not describing what the code does. 

// bad comment
// increment i by 1
i++;

// good comment  
// retry mechanism: exponential backoff prevents overwhelming the api
await sleep(Math.pow(2, retryCount) * 1000);

conclusion

writing clean code is a skill that develops over time. start with good naming, keep functions focused, and always think about the person who will read your code later—because that person is usually you, six months from now, at 2am, trying to fix a bug.

remember: code that works is good. code that works and is maintainable is great. code that works, is maintainable, and doesn't make people cry is art.
